# Literature Review Harness 答辩文档

## 1. 项目一句话介绍

本项目是一个面向学术 survey 生成的 agentic harness。系统不是直接让大模型自由写综述，而是先通过 Sciverse 检索、可选 MinerU 全文解析、运行时 Evidence KB、Skill 按需加载、多智能体审查和引用校验，构建一个可追溯的文献证据链，再生成带 evidence id 引用、图文展示和多格式导出的学术 survey。

更简洁地说：

> 这是一个 evidence-grounded survey harness：它把论文检索、证据构建、结构规划、技能调用、正文生成、审查修订、引用校验和图文导出组织成一个可审计的 agent 工作流。

## 2. 项目解决的问题

传统让 AI 写 survey 常见问题包括：

- 大模型容易编造论文、作者、结论和引用。
- 检索到的论文和最终正文之间缺少可追踪关系。
- 生成内容可能只是论文罗列，缺少 survey 需要的结构、taxonomy、比较和开放问题。
- 论文全文解析、摘要片段、检索结果、写作技能和图片生成通常是孤立模块，缺少统一调度。
- 最终输出很难被验证：不知道每个段落依据哪条文献证据。

本项目的设计目标是把 survey 写作拆成可控的 harness 流程：

1. 先构建文献证据库，而不是先写正文。
2. 每条证据有稳定 evidence id，例如 `P001-E01`。
3. 所有实质性段落必须引用 evidence id。
4. 引用校验器会拒绝未知 evidence id 和缺引用段落。
5. 多智能体审查器从内容质量、引用准确性、结构完整性三个角度审查草稿。
6. 图片生成和导出是后处理，不影响证据校验主链路。

## 3. 项目定位

本项目不是单纯的多智能体聊天系统，也不是普通的文献总结脚本。更准确的定位是：

> 一个状态驱动、证据驱动、工具增强的 Survey Harness。

其中：

- 状态驱动：`LiteratureKB` 保存论文、证据、解析全文和 MinerU 状态。
- 证据驱动：最终 survey 的引用来自 KB 中的 evidence id。
- 工具增强：Agent 通过 tool calling 调用检索、上下文读取、MinerU 解析、Skill 加载、survey context 规划等工具。
- 多智能体增强：生成后用三个 reviewer agent 并行审查。
- 可审计：输出 `evidence_pack.json`、`check_report.json`、`skill_trace.json`、`figure_plan.json`。

### 3.1 答辩命名口径

代码仓库名和 README 中仍使用 `Literature Review Harness`，但答辩时建议统一称为 `Survey Harness`。原因是本项目当前输出更接近 AI/CS 领域的 survey：重点是梳理领域方法、形成组织框架、比较技术路线、总结应用和开放问题，而不是医学或社会科学语境下严格的 systematic review。

推荐答辩名称：

> Taxonomy- and Evidence-Grounded Survey Harness

如果需要兼容仓库名，可以说：

> Our project is named Literature Review Harness in the repository, but its target output is an academic survey paper. In the defense, we frame it as a Survey Harness.

## 4. 总体架构

系统主链路如下：

```text
用户输入主题
  -> Config 读取环境变量和配置
  -> 初始化 LLMClient / LiteratureKB / ToolRegistry / SkillManager
  -> AgentLoop 构建 system prompt 和 tool schema
  -> LLM 调用 build_literature_kb
  -> Sciverse semantic/meta 检索，写入 Evidence KB
  -> 可选 MinerU 解析全文，补充 parsed text evidence
  -> LLM 调用 list_evidence / read_evidence / read_context / search_literature
  -> 按需调用 skill tools 发现、路由、加载、卸载写作技能
  -> 调用 prepare_survey_context 生成 timeline、citation map、outline、evidence needs
  -> LLM 在 AgentLoop 内生成完整 Markdown survey
  -> MultiAgentReviewer 三路并行审查
  -> CitationVerifier 强制引用校验
  -> 可选图片生成与 SVG fallback
  -> 导出 Markdown / HTML / LaTeX / evidence pack / check report / skill trace
```

可以用三层解释：

### 4.1 Literature Evidence Layer

负责建立可靠来源。

核心模块：

- `src/tools/literature_kb.py`
- `src/tools/mineru.py`
- `src/state/kb.py`

主要职责：

- 多源检索论文。
- 生成论文记录 `PaperRecord`。
- 生成证据记录 `EvidenceRecord`。
- 合并重复论文。
- 生成稳定 evidence id。
- 可选解析论文全文。
- 保存证据包供审计。

### 4.2 Agent Orchestration Layer

负责让模型按流程调用工具。

核心模块：

- `src/agent/context.py`
- `src/agent/loop.py`
- `src/tools/registry.py`
- `src/llm/client.py`

主要职责：

- 构建 system prompt。
- 暴露工具 schema。
- 调用 Intern-S2 OpenAI-compatible API。
- 解析 tool calls。
- 并行执行多个 tool call。
- 将工具结果写回 conversation。
- 根据 stop condition 决定是否继续修订。

### 4.3 Review, Validation and Output Layer

负责质量审查、引用校验、图片生成和导出。

核心模块：

- `src/validation/multi_agent.py`
- `src/validation/citations.py`
- `src/images/*`
- `src/exporters/*`
- `src/utils/runs.py`

主要职责：

- 多智能体审查内容质量、引用准确性、结构完整性。
- 强制最终正文引用 KB 中存在的 evidence id。
- 生成 figure plan。
- 生成图片或本地 SVG fallback。
- 导出 HTML、LaTeX。
- 写入 run directory 和 latest pointer。

## 5. 入口文件设计

入口文件是 `main.py`。

它主要做 8 件事：

1. 读取配置：`Config.from_env()`。
2. 创建本次运行目录：`create_run_paths(topic)`。
3. 初始化 LLM：`LLMClient(config.intern_api_base, config.intern_api_key, config.model)`。
4. 初始化运行时 KB：`LiteratureKB()`。
5. 注册工具：`ToolRegistry().register(...)`。
6. 初始化 Skill 系统：`SkillManager`、`SkillRouter`、`SkillTraceRecorder`。
7. 创建 AgentLoop，并加入两个 stop condition：
   - `MultiAgentReviewer`
   - `CitationVerifier`
8. 保存输出并导出：
   - `survey.md`
   - `survey.html`
   - `survey.tex`
   - `evidence_pack.json`
   - `check_report.json`
   - `skill_trace.json`
   - `figure_plan.json`

入口文件的价值在于它把所有模块组装成完整 harness，而不是把逻辑全部写在一个大函数中。

## 6. 核心数据结构

### 6.1 PaperRecord

位置：`src/state/kb.py`

表示一篇论文。

字段包括：

- `paper_id`：稳定论文 ID，例如 `P001`。
- `title`：论文标题。
- `year`：年份。
- `venue`：发表 venue。
- `doi`：DOI。
- `authors`：作者列表。
- `doc_id`：Sciverse 文档 ID。
- `parse_url`：可用于 MinerU 解析的 PDF/HTML URL。
- `mineru`：MinerU 状态字典。
- `evidence_ids`：该论文下所有证据 ID。

### 6.2 EvidenceRecord

表示一条可引用证据。

字段包括：

- `evidence_id`：稳定证据 ID，例如 `P001-E01`。
- `paper_id`：所属论文。
- `title`：论文标题。
- `text`：证据文本，截断到 1600 字符以内。
- `source`：证据来源，例如 `sciverse_semantic`、`sciverse_metadata_abstract`、`mineru_structured`。
- `doc_id`：Sciverse 文档 ID。
- `offset`：上下文位置。
- `score`：检索得分。
- `year`：论文年份。

### 6.3 ParsedDocument

表示 MinerU 解析出的全文。

字段包括：

- `paper_id`
- `title`
- `text`
- `source`
- `metadata`

全文不会直接全部塞进 prompt，而是通过 `read_parsed_paper` 和 `search_parsed_paper` 按需读取，并在读取时转化为新的 citeable evidence。

## 7. LiteratureKB 的作用

`LiteratureKB` 是整个系统的运行时知识库。

它负责：

- 管理论文列表。
- 管理 evidence 列表。
- 管理 parsed document。
- 通过 DOI、doc_id、title 合并重复论文。
- 自动分配 `P001`、`P001-E01` 这类稳定 ID。
- 为 CitationVerifier 提供合法 evidence id 集合。
- 输出 `evidence_pack.json`。

关键方法：

- `upsert_paper()`：新增或合并论文。
- `add_evidence()`：新增 evidence，生成 evidence id。
- `get_evidence()`：根据 evidence id 读取证据。
- `set_parsed_document()`：保存 MinerU 全文解析结果。
- `keep_first_papers()`：限制论文数量并重建索引。
- `to_json()`：输出审计包。

答辩时可以强调：

> LiteratureKB 是本项目避免幻觉的基础。最终 survey 中引用的不是模型自由生成的引用，而是 KB 中已经存在的 evidence id。

## 8. 工具系统设计

工具系统由 `ToolRegistry` 管理。

每个工具需要满足：

```python
name: str
description: str
parameters: dict
async execute(...)
```

`ToolRegistry` 的职责：

- 注册工具。
- 导出 OpenAI function-calling schema。
- 根据 LLM tool call 执行对应工具。
- 捕获工具异常。
- 支持 post-execute hook。

当前主工具包括：

| 工具 | 作用 |
|---|---|
| `build_literature_kb` | 构建文献 evidence KB，是写作前必须调用的主工具 |
| `list_evidence` | 列出 KB 中已有 evidence |
| `read_evidence` | 读取单条 evidence 全文 |
| `search_literature` | 追加 Sciverse semantic 检索结果 |
| `read_context` | 读取 Sciverse content API 上下文并写入 citeable evidence |
| `list_parsed_papers` | 查看 MinerU 已解析全文的论文 |
| `read_parsed_paper` | 按 offset 读取 MinerU 全文片段并写入 evidence |
| `search_parsed_paper` | 在 MinerU 全文中搜索关键词并写入 evidence |
| `prepare_survey_context` | 构造 timeline、citation map、outline 和 writing plan |
| `skills_*` | 发现、路由、加载、读取、卸载外部 skill |

## 9. Sciverse 检索设计

Sciverse 相关逻辑主要在 `src/tools/literature_kb.py`。

系统使用两类 Sciverse API：

1. `agentic-search`
   - 用于 semantic search。
   - 返回相关论文片段。
   - 写入 `sciverse_semantic` evidence。

2. `meta-search`
   - 用于补充 metadata 和 abstract。
   - 若 abstract 足够长，写入 `sciverse_metadata_abstract` evidence。

检索结果会经过 metadata 归一化：

- title
- abstract
- doc_id
- doi
- year
- venue
- authors
- open access URL
- arXiv PDF URL 推断

系统会根据 DOI、doc_id、title 合并论文，避免重复记录。

## 10. MinerU 解析设计

MinerU 是增强源，不是硬依赖。

位置：`src/tools/mineru.py`

核心策略：

- 只解析有可解析 URL 的论文。
- 支持 PDF、HTML、doc、ppt 等。
- 默认快模式只取 `1-12` 页。
- 默认关闭 table/formula，以提高速度。
- batch size 默认是 1。
- 超时不会阻塞主链路。
- 失败或无 token 时自动退化为 Sciverse-only。

解析成功后：

1. 从 zip 中优先读取 `content_list.json` 或 `content_list_v2.json`。
2. 如果没有 structured JSON，则尝试读取 `full.md`。
3. 截断 References 之后的内容，避免参考文献污染证据。
4. 生成若干 `mineru_structured` evidence。
5. 保存完整 parsed document，供后续按需读取。

答辩时可以这样解释：

> MinerU 不作为硬依赖，是为了保证系统鲁棒性。即使全文解析失败，Sciverse 证据仍然可以支撑 survey 生成；如果 MinerU 成功，则能为关键方法细节、实验设定和局限性提供更细粒度证据。

## 11. AgentLoop 工作机制

位置：`src/agent/loop.py`

AgentLoop 是核心执行循环。

每轮步骤：

1. 从 ContextBuilder 获取 messages 和 tools。
2. 调用 LLM。
3. 如果 LLM 返回 tool calls：
   - 提取工具名和 JSON 参数。
   - 如果有多个工具调用，则并行执行。
   - 把工具结果作为 `role=tool` 消息写回对话。
   - 进入下一轮。
4. 如果 LLM 返回最终正文：
   - 运行 stop conditions。
   - 如果审查或引用校验失败，把反馈作为 user message 写回。
   - 让 LLM 继续修订。
   - 如果全部通过，返回最终正文。
5. 达到最大轮数则停止。

这个循环体现了 harness 的关键能力：

- 工具调用控制。
- 并行工具执行。
- 运行时状态积累。
- 质量条件驱动的自修订。
- 最终结果不是一次生成，而是经过审查和校验。

## 12. ContextBuilder 和 System Prompt

位置：`src/agent/context.py`

System prompt 规定了系统行为：

- 必须先调用 `build_literature_kb`。
- 必须使用 evidence 工具查看证据。
- Sciverse 片段不够时使用 MinerU parsed paper 工具。
- Skill 是写作协议，不是事实来源。
- 必须调用 `prepare_survey_context`。
- 最终直接输出完整 Markdown survey。
- 每个实质性段落必须引用 evidence id。
- 不能编造论文、作者、结论。
- References 只能列出被 evidence id 引用过的论文。

ContextBuilder 也提供扩展点：

- `pre_llm_hooks`
- `post_tool_hooks`

后续如果要做工具过滤、上下文压缩、动态 prompt 注入，可以从这里扩展。

## 13. Survey Context 规划

位置：`src/tools/survey_context.py`

`prepare_survey_context` 是正文生成前的结构规划工具。

它生成：

- `coverage`：论文数量、证据数量、年份范围、论文列表。
- `timeline`：按年份排序的论文时间线。
- `citation_map`：每条 evidence 可支撑的 claim candidate。
- `survey_design`：由 LLM 生成的 selected papers、low relevance papers、evidence needs、recommended outline、writing plan。

它不是写作工具，而是规划工具。

设计中的防幻觉点：

- 只允许使用 payload 中已有 paper id。
- 过滤掉 LLM 编造的 paper id。
- `evidence_needs.suggested_tools` 只能来自允许列表：
  - `search_literature`
  - `read_context`
  - `list_parsed_papers`
  - `read_parsed_paper`
  - `search_parsed_paper`
- 输出 outline 前会做清洗。

## 14. Skill System 设计

Skill 系统是项目的扩展机制。

核心文件：

- `src/skill_system/manager.py`
- `src/skill_system/router.py`
- `src/skill_system/tools.py`
- `src/skill_system/trace.py`

它的核心思想是 progressive disclosure：

> 先只读取 skill metadata；只有当 agent 明确需要某个阶段能力时，才加载完整 skill 内容；使用后立即 unload。

这样做的原因：

- 避免一次性把大量 skill 文档塞进上下文。
- 降低 token 成本。
- 避免无关 skill 干扰模型。
- 让 skill 使用过程可审计。

内置或可配置角色包括：

- `research_framing`
- `survey_writing`
- `citation_grounding`
- `self_review`
- `academic_polishing`
- `figure_planning`
- `figure_generation`
- `figure_verification`
- `latex_arxiv_export`

Skill 工具包括：

- `skills_list_index`
- `skills_route_for_phase`
- `skills_load_for_phase`
- `skills_resource_index`
- `skills_read_resource`
- `skills_run_script`
- `skills_unload`

审计输出：

- `skill_trace.json`

答辩时可以强调：

> Skill 系统不是把外部材料当事实来源，而是把它们当写作协议、引用规范、图表规划指导和润色规范。事实仍然必须来自 Evidence KB。

## 15. 多智能体审查机制

位置：`src/validation/multi_agent.py`

系统在最终正文生成后并行运行三个 reviewer agent：

1. `content_quality`
   - 检查 survey 是否有清晰 scope 和 organizing argument。
   - 检查是否只是论文罗列。
   - 检查方法、tradeoff、limitations 是否足够具体。

2. `citation_accuracy`
   - 检查是否使用 `Pnnn-Enn` evidence id。
   - 检查 citation 是否分布合理。
   - 检查表格事实行是否有 evidence。
   - 检查 References 是否和 evidence id 对应。

3. `structure_completeness`
   - 检查 Abstract、Introduction、taxonomy/framework、comparison、future agenda、conclusion、references 等结构。
   - 检查章节是否平衡。
   - 检查摘要和结论是否有 survey 风格。

三个 reviewer 并行执行，聚合反馈后写回 AgentLoop。

如果审查失败：

- MultiAgentReviewer 返回 `False`。
- AgentLoop 把审查意见作为 user message。
- 主 agent 继续修订完整 survey。

如果审查通过：

- 进入 CitationVerifier。

## 16. CitationVerifier 引用校验

位置：`src/validation/citations.py`

CitationVerifier 是硬规则校验器，不依赖 LLM 判断。

它检查：

- 是否有空 evidence KB。
- 是否出现未知 evidence id。
- 是否出现模型错误响应。
- 每个实质性段落是否包含 evidence id。
- References 之前的正文是否符合引用要求。

它会跳过：

- 标题段落。
- 代码块。
- 图片。
- figure HTML。
- Markdown 表格。
- References 标题。
- 少于 80 字的短段落。

如果失败，会把错误写回 conversation，让 LLM 继续修订。

这也是本项目最重要的可靠性机制之一。

## 17. 图片生成与图文并茂

图片生成是可选后处理，位置在 `src/images/`。

设计原则：

- 图片不参与事实校验。
- 事实性 claim 仍然必须由正文 evidence id 支撑。
- 图片 prompt 不要求画出精确论文题名、作者、DOI 或 citation id。
- 图片 caption 会保留 source evidence ids。
- 图片失败不阻塞 survey 正文生成。

主要模块：

### 17.1 planner.py

负责从最终 Markdown 中解析章节，选择适合插图的 section。

可规划的图类型：

- conceptual overview
- method taxonomy
- comparison matrix
- research agenda

规划逻辑：

- 只选择已经包含 evidence id 的章节。
- 最多生成 3 张图。
- 根据章节标题选择图类型。
- 图片计划写入 `figure_plan.json`。

### 17.2 generator.py

封装 OpenAI Images API。

支持：

- base_url 可配置。
- endpoint_path 可配置。
- 读取 `b64_json` 或 URL 图片。
- 输出 `GeneratedImage` metadata。

### 17.3 vector.py

本地 SVG fallback。

如果 raster image 生成失败，系统可以生成：

- taxonomy SVG
- comparison matrix SVG
- research agenda SVG

这保证了图文功能具备鲁棒性。

### 17.4 pipeline.py

图片生成总控：

- 根据配置判断是否启用。
- 生成 figure plan。
- 调用 OpenAI image generator 或本地 SVG。
- 写入 manifest。
- 把图片插入 survey 对应章节。

## 18. 导出系统

导出模块包括：

- `src/exporters/html.py`
- `src/exporters/latex.py`

### 18.1 HTML 导出

功能：

- 将 Markdown 转成科研风格 HTML。
- 支持标题、段落、列表、图片、figure、figcaption。
- evidence id 会高亮成 `<code class="evidence-id">`。
- 输出页面有论文式排版。

### 18.2 LaTeX 导出

功能：

- 将 Markdown 转成基础 LaTeX。
- 支持标题映射为 section/subsection。
- 支持图片 figure 环境。
- 支持 LaTeX 特殊字符转义。
- 输出 `survey.tex`。

导出系统当前是基础版本，适合答辩演示和后续扩展。

## 19. 输出目录设计

每次运行会生成独立 run directory：

```text
output/runs/YYYYMMDD-HHMM-topic/
  survey.md
```

## 22. 项目创新点

### 22.1 Evidence-first，而不是 generation-first

系统要求先构建 Evidence KB，再写 survey。最终正文必须引用 KB 中的 evidence id。

这和普通“输入主题然后直接生成文章”的方式不同。

### 22.2 Stable Evidence ID

系统把论文和证据统一编号：

```text
P001
P001-E01
P001-E02
```

这些 ID 同时出现在：

- survey 正文。
- evidence pack。
- check report。
- figure caption。

因此答辩时可以点击 evidence id 反查来源。

### 22.3 LLM + deterministic verifier

系统不是完全依赖 LLM 自我检查。

多智能体审查是 LLM-based。

CitationVerifier 是 deterministic rule-based。

二者结合：

- LLM reviewer 负责语义质量。
- deterministic verifier 负责硬约束。

### 22.4 Progressive Skill Loading

Skill 不会一次性全部加载。

系统先路由，再加载，再卸载。

这样能减少上下文污染，也让 skill 使用过程可审计。

### 22.5 Opportunistic MinerU

MinerU 成功时提供全文增强。

MinerU 失败时不影响主链路。

这种设计适合真实场景，因为学术 PDF 解析经常遇到失败、超时、无权限等问题。

### 22.6 图文后处理和安全 fallback

图片生成不是主链路依赖。

没有 OpenAI image key 或图片生成失败时，仍然可以 fallback 到本地 SVG。

## 23. 和 LangGraph / Harness 思想的关系

这个项目目前没有直接依赖 LangGraph，但架构思想与 LangGraph-style harness 接近：

- 有明确状态：`LiteratureKB`。
- 有工具节点：Sciverse、MinerU、Skill、SurveyContext。
- 有循环调度：`AgentLoop`。
- 有停止条件：`MultiAgentReviewer` 和 `CitationVerifier`。
- 有可审计中间产物：evidence pack、skill trace、check report。

如果后续重构，可以把当前流程映射为 LangGraph：

```text
BuildKBNode
  -> EvidenceGatherNode
  -> SkillRouteNode
  -> SurveyContextNode
  -> DraftNode
  -> MultiAgentReviewNode
  -> CitationVerifyNode
  -> ImagePlanNode
  -> ExportNode
```

当前项目相当于一个轻量自研 agent loop 版本。

## 26. 可能被问到的问题与回答

### Q1：这个系统和普通 RAG 有什么区别？

普通 RAG 通常是检索文档片段后直接生成回答。本项目不仅检索，还把论文、证据、parsed document、skill trace、check report 组织成可审计的 survey harness。最终正文必须引用稳定 evidence id，并经过多智能体审查和 deterministic citation verifier。

### Q2：为什么需要 Evidence KB？

Evidence KB 是防幻觉的核心。大模型生成的每个实质性段落必须引用 KB 中已有 evidence id，CitationVerifier 会拒绝未知 ID 或缺引用段落。这样可以把“模型说了什么”和“证据来自哪里”连接起来。

### Q3：如果 Sciverse 检索不全怎么办？

系统支持多轮 `search_literature` 和 `read_context`，并且 `prepare_survey_context` 会输出 evidence needs，提示缺少哪些证据。未来可以接入 arXiv、Semantic Scholar、OpenAlex 等更多检索源。

### Q4：MinerU 失败会不会导致系统失败？

不会。MinerU 是 opportunistic enhancement。失败、超时、无 token 都会自动降级为 Sciverse-only。系统会在 `evidence_pack.json` 里记录 MinerU 状态。

### Q5：多智能体在哪里？

主要体现在生成后的并行审查。系统有三个 reviewer agent，分别检查内容质量、引用准确性和结构完整性。它们不是自由聊天，而是作为 stop condition 嵌入 AgentLoop。

### Q6：为什么不直接让多智能体互相讨论写论文？

自由讨论式多智能体容易状态不稳定、证据难追踪、成本高。我们采用 harness 方式：状态由 KB 管理，工具由 registry 管理，审查由 stop condition 管理，最终结果由 verifier 硬校验。

### Q7：Skill 系统解决什么问题？

Skill 系统提供研究构思、survey 写作、citation grounding、图表规划等协议化能力。它通过 metadata routing 和 progressive disclosure 控制上下文注入，避免一次性加载大量无关技能。

### Q8：怎么证明引用没有乱编？

系统有三重机制：

1. System prompt 要求只引用 evidence id。
2. CitationVerifier 检查 evidence id 是否存在。
3. `evidence_pack.json` 可反查每个 evidence id 的来源文本。

### Q9：为什么图片生成不进入 citation verifier？

图片是表达层，不作为事实来源。事实 claim 必须在正文里由 evidence id 支撑。图片 caption 会记录 source evidence ids，但图片生成失败不会影响正文证据链。

### Q10：系统目前有哪些局限？

主要局限：

- 目前主流程依赖 Intern-S2 和 Sciverse API。
- LaTeX 导出是基础版，还不是期刊模板级。
- CitationVerifier 只能检查 evidence id 存在和段落引用，不能完全判断“该 evidence 是否真的支持该句 claim”。
- 当前 AgentLoop 是自研轻量循环，还不是完整 LangGraph 状态机。
- HTML/LaTeX 的 Markdown 支持较基础。

### Q11：下一步如何改进？

可改进方向：

- 接入更多检索源：Semantic Scholar、OpenAlex、arXiv。
- 把流程迁移到 LangGraph，支持节点级 checkpoint 和可视化状态。
- 增加 claim-evidence entailment verifier，判断引用是否真正支持句子。
- 生成 method taxonomy、comparison matrix、timeline 等更明确的 survey 结构。
- 改进 LaTeX 导出，支持 ACM/IEEE/arXiv 模板。
- 增加 Web UI，让用户交互式筛选论文和查看 evidence。

## 28. 最终答辩表述

可以用下面这段作为答辩开场：

> 我们的项目是一个面向学术 survey 生成的 evidence-grounded agent harness。和普通大模型写作不同，我们先通过 Sciverse 和可选 MinerU 构建 Literature Evidence KB，为每篇论文和每条证据分配稳定 ID；随后 agent 通过工具调用读取证据、加载写作 skill、生成 survey context，并在证据约束下完成正文写作。最终草稿会经过三个 reviewer agent 的并行审查，以及 deterministic citation verifier 的硬校验。系统输出 Markdown、HTML、LaTeX、evidence pack、check report 和 skill trace，从而实现从检索、证据、写作到审计的完整闭环。

## 29. 答辩时要避免的说法

不建议说：

- “我们就是调用大模型写综述。”
- “我们保证所有引用语义完全正确。”
- “MinerU 一定会解析所有论文。”
- “图片里的内容就是证据。”
- “Skill 是事实来源。”

建议说：

- “我们通过 evidence id 降低引用幻觉。”
- “系统提供可审计证据链。”
- “MinerU 是增强源，不是硬依赖。”
- “图片是表达层，事实仍由 evidence 支撑。”
- “Skill 是写作协议和方法指导。”

## 30. 一句话结论

本项目的核心价值不是“生成一篇 survey”，而是建立一条从论文检索、证据构建、结构规划、写作生成、审查修订到引用校验和多格式导出的可审计 survey harness。
